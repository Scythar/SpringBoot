✅ What is restAPI?
ANSWER:
REST = Representational State Transfer
API = Application Programming Interface

Representational State Transfer (REST): This refers to the concept that when you request a resource (like a user or an image), the server "transfers" a "representation" of the current "state" of that resource (usually in JSON or XML format) to your machine.

Application Programming Interface (API): This is the "contract" or set of rules that allows two pieces of software to talk to each other.

✅ Difference Between @Controller and @RestController
ANSWER:

@Controller
Used mainly for MVC / web applications
Returns:
View name (JSP / Thymeleaf)
Needs @ResponseBody annotation if you want JSON

Example:
@Controller
public class UserController {
   @ResponseBody
   @GetMapping("/user")
   public User getUser() {
       return new User();
   }
}


@RestController
Used for REST APIs
Automatically converts returned object to JSON
No need for @ResponseBody annotation
Equivalent to:
@Controller
@ResponseBody


🔥 What Happens Internally in @RestController?
1️⃣ Controller method returns an object
return new OrderResponseDTO();
2️⃣ Spring uses HttpMessageConverters
Jackson (JSON) is one of them and default used by Spring
Converts Java object → JSON
3️⃣ Response is written to HttpServletResponse body
Content-Type: application/json
Status: 200 OK (default)
4️⃣ DispatcherServlet sends response back
Client receives JSON.
🔍 Important Interview Detail
If Jackson is not present in classpath:
Spring cannot convert object → JSON
You’ll get HTTP 500 error
⭐ Quick Interview Summary Line (Memorize This)
"@RestController automatically serializes the return object to JSON using HttpMessageConverters like Jackson, whereas @Controller is used for view-based MVC."

✅What is the difference between @RequestBody and @ModelAttribute?
@RequestBody is used for REST APIs to deserialize JSON payloads, while @ModelAttribute is used in MVC to bind form and request parameters to a model object.

@RequestBody
Reads raw HTTP request body
Uses HttpMessageConverters (Jackson)
Converts JSON → Java object

Example:
@PostMapping("/users")
public UserDTO create(@RequestBody UserDTO dto) { }
Request:
{
  "name": "Nitesh",
  "email": "nitesh@gmail.com"
}
✔ Used in REST APIs
✔ JSON / XML payload
✔ One-time deserialization


@ModelAttribute
Used mainly for MVC / web applications
View name (JSP / Thymeleaf)
Binds data from:
Form data
Query parameters
Path parameters
Uses Spring’s Data Binder
Field-by-field binding

Example:
@PostMapping("/register")
public String register(@ModelAttribute UserForm form) { }
Form submission:
name=Nitesh&email=nitesh@gmail.com
✔ Used in MVC / form-based apps
✔ Works even without JSON
✔ Supports validation & default values

✅What is the difference between PUT and PATCH?
PUT replaces the entire resource and is idempotent, whereas PATCH performs partial updates and is safer when only a few fields need to change.

PUT
Replaces the entire resource
Client sends full representation
Missing fields may be overwritten or null
Idempotent (calling it multiple times gives same result)
Example:
PUT /users/10
{
  "name": "Nitesh",
  "email": "nitesh@gmail.com",
  "age": 28
}
✔ Used when client knows complete state
✔ Common in internal APIs

If some fields are missing in the JSON request body:
No error is thrown
Missing fields are set to:
null for objects and 0 / false for primitives


PATCH
Updates only specific fields
Client sends partial data
Not guaranteed to be idempotent
Safer for partial updates
Example:
PATCH /users/10
{
  "email": "newmail@gmail.com"
}
✔ Used when updating few fields only
✔ Prevents accidental data loss

✅What is idempotency? Give real API examples where idempotency matters.

ANSWER:
Idempotency means performing the same operation multiple times produces the same final state as performing it once.

This matters because retries, network failures, and duplicate events are normal in real systems.

PUT API (Idempotent)
PUT /users/10
{
  "email": "a@gmail.com"
}
Call once → email = a@gmail.com
Call 10 times → still a@gmail.com

⭐ 2. How to Implement Idempotency
Use an Idempotency Key
Client sends a unique key with each operation:
idempotency-key: 12345
Server stores:
request → response mapping
If same key arrives again → return stored result.
Used in: Stripe, Razorpay, PayPal

✅Which HTTP methods are idempotent and which are not?
Idempotent HTTP Methods
HTTP Method	Idempotent	Reason
GET	✅ Yes	Only reads data, no state change
PUT	✅ Yes	Replaces entire resource with same state
DELETE	✅ Yes	Deleting again has no further effect
HEAD	✅ Yes	Same as GET but no body
OPTIONS	✅ Yes	Metadata only

Non-Idempotent HTTP Methods
HTTP Method	Idempotent	Reason
POST	❌ No	Creates new resource each time
PATCH	❌ Not guaranteed	Depends on implementation

✅What happens internally when you use @AuthenticationPrincipal UserDetails userDetails in a controller method?

ANSWER:
1️⃣ During request processing, your JWT filter validates the token and sets authentication:
SecurityContextHolder.getContext().setAuthentication(authentication);

2️⃣ The Authentication object contains:
principal → UserDetails
authorities
isAuthenticated = true

3️⃣ Spring MVC has a HandlerMethodArgumentResolver that:
Detects @AuthenticationPrincipal
Fetches the Authentication object from SecurityContextHolder
Extracts the principal
Injects it into the controller method parameter
UserDetails userDetails =
   (UserDetails) SecurityContextHolder
       .getContext()
       .getAuthentication()
       .getPrincipal();
This is done automatically by Spring.

🔥 Why this is useful (Real Reason)
✔ Avoids extra DB calls
✔ Gives authenticated user context instantly
✔ Clean controller code
✔ Works across all endpoints after authentication


⚠ Important Interview Edge Case
If:
User is unauthenticated OR
JWT is missing/invalid
Then:
@AuthenticationPrincipal UserDetails userDetails
→ will be null
